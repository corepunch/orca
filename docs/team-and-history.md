# Team, Skills & Project History

This page serves two purposes:

1. **Recommended agent team** — the roles and skills an AI coding agent team needs to be effective on the ORCA codebase.
2. **Project history** — a chronological summary of every major feature area implemented, how each was done, and how it is used today.

---

## Recommended Agent Team

ORCA spans C engine code, an OpenGL renderer, a PHP code-generation toolchain, Lua scripting, cross-platform builds, and a full MkDocs documentation site. No single agent is fluent in every layer. The table below lists the natural team split.

### Core Roles

| Role | Primary Skills | Key Files |
|---|---|---|
| **C Engine Agent** | C99, data structures, FNV1a hashing, linked lists, memory management | `source/core/`, `include/orca.h` |
| **Renderer Agent** | OpenGL 3.3+, GLSL ES, texture formats, stencil/framebuffer, font rasterization (FreeType) | `source/renderer/`, `include/renderer.h` |
| **Toolchain Agent** | PHP (pyphp subset), Python, XML/DTD, Makefile, code generation patterns | `tools/`, `docs/schemas/module.dtd` |
| **Lua / UIKit Agent** | Lua 5.4 C API, metatables, component XML, UIKit layout system | `plugins/UIKit/`, `source/core/core_export.c` |
| **Platform / Build Agent** | Make, premake5, LuaRocks, Emscripten/WebGL, GitHub Actions CI | `Makefile`, `Makefile.webgl`, `.github/workflows/` |
| **Documentation Agent** | Markdown, MkDocs Material, DTD schema authoring | `docs/`, `mkdocs.yml` |

### Skills That Span Multiple Roles

The following skills are needed by more than one role and should be considered table stakes for any agent working in this repository.

| Skill | Why it matters |
|---|---|
| **Module XML format** | Every C module exposes its public API through a `.xml` file; all agents need to read and sometimes extend it |
| **FNV1a hashing** | Property and event names are hashed to `uint32_t` at build time; agents must understand how IDs are generated |
| **Object/Component dispatch model** | All engine behaviour flows through `ObjProc` callbacks; agents need the mental model to add behaviour correctly |
| **pyphp template limitations** | Several standard PHP constructs are unsupported (see [Code Generation](architecture/code-generation.md#pyphp-limitations)); toolchain agents must know these before editing templates |
| **Git submodule workflow** | `libs/platform` and `tools/pyphp` are submodules; every agent must run `git submodule update --init --recursive` before building |

## Project History

The sections below trace the major implementation milestones in the repository, from the first commit to the present. Each section explains **what was built**, **how it was implemented**, and **how it is used today**.

---

### 1. Initial Engine Foundation

**Commits:** `52d71c5` (initial fresh commit) → `ba87262` (SceneKit added)

**What was built:**  
The ORCA core engine: an Object/Component system written in C, an OpenGL renderer, a Lua 5.4 scripting host, an XML filesystem, and the basic UIKit plugin with `Button`, `Label`, `StackView`, and `Screen`.

**How it was implemented:**  
- Objects are `malloc`'d structs with a name, a parent pointer, a children linked list, and a flat byte buffer (`data[]`) for component property storage.
- Components are described by `ClassDesc` and registered with `OBJ_RegisterClass()`. They receive messages (`kEventCreate`, `kEventDraw`, …) via an `ObjProc` callback.
- The renderer builds `ViewEntity` descriptors on the stack each frame and submits them to `R_DrawEntity`.
- Lua bindings are generated by a Python script (`conv-module.py`) from XML module definitions.

**How it is used today:**  
Every UI element in every ORCA application is an Object with UIKit components attached. The C Object/Component core described here has not fundamentally changed — later work has only added features on top of it.

### 2. LuaRocks Packaging

**PR:** #10 — `96d8265` → `be1d50a`

**What was built:**  
ORCA as an installable LuaRocks package. Adds `orca-dev-1.rockspec`, adjusts `Makefile` for external `CFLAGS`/`LDFLAGS`, sets up RPATH, and documents installation in `LUAROCKS.md`.

**How it was implemented:**  
The rockspec lists external C libraries (`libxml2`, `freetype2`, `lua5.4`, libjpeg, libpng, zlib) via `pkg-config`. `make install` copies the compiled `.so` and Lua modules into the LuaRocks tree.

**How it is used today:**  
```sh
luarocks install orca
```
After installation, `require "orca"` loads the engine without needing a manual build.

### 3. GitHub Pages Documentation Site

**PR:** #9 — `076f478` → `a282de5`

**What was built:**  
A MkDocs Material documentation website deployed automatically to GitHub Pages on every push to `main`.

**How it was implemented:**  
- `mkdocs.yml` configures MkDocs Material with a dark/light palette toggle.
- A GitHub Actions workflow (`.github/workflows/`) runs `mkdocs build` and deploys `site/` to the `gh-pages` branch.
- A DTD schema (`docs/schemas/orca.dtd`) is committed and published so editors can use it for XML autocomplete via `<!DOCTYPE … SYSTEM "https://corepunch.github.io/orca/schemas/orca.dtd">`.

**How it is used today:**  
The live site at `https://corepunch.github.io/orca/` is the primary reference for users and contributors. The DTD URL in the site is embedded in every example XML file.

### 4. UTF-8 / Unicode Support in JSON Parser

**PR:** #2 — `e48c0de`

**What was built:**  
Proper `\uXXXX` Unicode escape decoding in `p_json`, emitting valid multi-byte UTF-8 sequences instead of silently dropping non-ASCII characters.

**How it was implemented:**  
The parser calls `sscanf(…, "%4hx", &codepoint)` to decode the hex escape, then encodes the codepoint as 1–4 bytes of UTF-8 using a helper function (1 byte for ASCII, 2–3 bytes for the Basic Multilingual Plane, 4 bytes for supplementary planes). Surrogate pair detection and bounds checking were added.

**How it is used today:**  
Any JSON data loaded through the engine (localization tables, config files) correctly handles non-ASCII text, including CJK characters and emoji.

---

### 5. Lua Module Auto-Loading (`orca.__index`)

**PR:** #22 — `73a5ef5` → `e429bf4`

**What was built:**  
A `__index` metamethod on the `orca` global table that automatically `require`s `orca.<key>` on first access and caches the result, so `orca.geometry` and `require "orca.geometry"` are equivalent.

**How it was implemented:**  
The metamethod is registered in `source/core/orca.c` under an `EASY_MODULE_ACCESS` compile-time flag. It calls `lua_getfield(L, LUA_REGISTRYINDEX, "package")`, resolves `package.preload["orca.<key>"]` or `require`, and stores the result in the `orca` table for future lookups.

**How it is used today:**  
Lua scripts can write `orca.geometry.vec3(1, 0, 0)` without an explicit `require` line. All built-in submodules (`core`, `geometry`, `filesystem`, `UIKit`, …) are pre-registered in `package.preload` during engine startup.

### 6. Text Runs in TextBlock

**PR:** #14 / #17 — `a77d566` → `01b8dd8`

**What was built:**  
Multi-run text support in `TextBlock` and `r_font.c`, allowing mixed fonts, sizes, and styles in a single text block with correct kerning across run boundaries.

**How it was implemented:**  
- A `ViewTextRun` struct was introduced, holding a font face, size, colour, and the text string slice.
- `MakeText` builds an array of up to `MAX_TEXT_RUNS` runs from the TextBlock's children (each child can specify different font attributes).
- The font rasterizer was refactored to accept a `TextRun[]` and apply per-run metrics while maintaining a single baseline and accounting for cross-run kerning with FreeType.

**How it is used today:**  
TextBlock supports bold/italic spans, colour runs, and link highlights by nesting child elements with different font attributes. The renderer merges them into one continuous laid-out line.

### 7. Code Generation Refactor (pyphp + PHP Templates)

**PRs:** #46, #47, multiple follow-up PRs — `1478954` → current `tools/`

**What was built:**  
A complete rewrite of the Python-based `conv-module.py` code generator into a PHP template architecture running through the `pyphp` Python-PHP bridge. Outputs are split into focused sub-templates orchestrated by controller classes.

**How it was implemented:**  
- `tools/pyphp/` (git submodule) provides a Python interpreter that can execute PHP scripts.
- `tools/model/module.php` defines the `Model` class that parses a `.xml` module file into `Interface`, `Struct`, `Enum`, `Component`, and `Method` objects.
- `tools/templates/controllers/CodegenController.php` orchestrates six header sub-templates and seven export sub-templates.
- `tools/templates/index.php` routes `"ctrl/action"` strings to controller methods.

**How it is used today:**  
```bash
cd tools && make          # regenerate all headers + export files
cd tools && make docs     # regenerate API Markdown docs
cd tools && make dtd      # regenerate orca.dtd
```
Adding a new module only requires writing a `.xml` file and running `make`. Generated files (`.h`, `_properties.h`, `_export.c`) are committed alongside source and must not be edited by hand.

### 8. Mesh Pointer Boxing (`BOX_PTR`)

**PRs:** #39, #40 — `eae29d2` → `9996228`

**What was built:**  
A scheme for selecting built-in shaders and mesh types via small integer tags cast to pointers, eliminating runtime lookups for the common case.

**How it was implemented:**  
`BOX_PTR(Type, TAG)` stores `TAG` (a small integer) directly in the pointer field. `BOX_IS_PTR(ptr)` tests whether the field holds a real heap address or a boxed tag. `R_DrawEntity` checks this before deciding how to resolve the mesh or shader.

**How it is used today:**  
Component draw code writes `entity.shader = BOX_PTR(Shader, SHADER_UI)` to select the UIKit shader without looking up a pointer. See [Mesh Pointer Boxing](MESH_POINTER_BOXING.md) for the full reference.

### 9. WebGL / Emscripten Build

**PRs:** #120 – #179 — `7a80abf` → `cc584eb`

**What was built:**  
A full WebGL 2.0 build of the ORCA engine running in the browser via Emscripten. Includes a progress-bar loading shell, a console overlay, iOS/Safari touch fixes, and a GitHub Actions CI job that deploys a live demo to GitHub Pages.

**How it was implemented:**  
- `Makefile.webgl` compiles every C source with `emcc`, links Lua 5.4 / libxml2 / liblz4 pre-built WASM libraries, and bundles the `share/` asset directory with `--preload-file`.
- `ASYNCIFY=1` is enabled: the Lua main loop calls `emscripten_sleep(0)` when idle, yielding control back to the browser's event loop.
- Plugins are compiled into the WASM binary and pre-registered via an auto-generated `plugins_luaopen.h` (no dynamic `.so` loading in WebGL).
- `web/shell.html` provides the loading spinner, canvas scaling, and a corner log overlay. A `snapCanvas` JS function snaps the canvas CSS dimensions to integers on iOS to prevent SAFE_HEAP misalignment crashes.
- GLSL ES compatibility patches: `GL_TEXTURE_SWIZZLE_*` gated behind `R_USE_SRGB`, Alpha8 textures expanded to RGBA.

**How it is used today:**  
```bash
make webgl          # build the WASM binary + HTML shell
```
The live demo at `https://corepunch.github.io/orca/demo/orca.html` is automatically deployed by CI on every push to `main`. Use `make webgl-debug` for a debug build with full stack traces.

### 10. DarkReign Plugin (SPR Format & Palette Rendering)

**Commits:** `dcb0f4f` (FtgPackage rewrite) and related SprFormat work

**What was built:**  
Support for the Dark Reign (Auran, 1997) `.spr` sprite format and `.pz2` package files, including palette-indexed rendering via a dedicated GPU shader.

**How it was implemented:**  
- `plugins/DarkReign/SprFormat.c` parses the `RSPR`/`SSPR` binary format (int32 header, pic-index table, RLE-compressed pixel data).
- Shadow sprites (`SSPR`) fill all opaque pixels with palette index 47 instead of reading pixel data.
- A 256-entry `palette_t` (`uint32_t[256]`, `0x00RRGGBB`, index 0 always transparent) is uploaded to the `TX_SPRITEPALETTE` texture slot via `R_GetSpritePalette()`. This is a separate slot from `TX_CINEMATICPALETTE`, which is used by the cinematic video player and the general `ViewEntity.palette` path.
- The built-in `SHADER_SPRITE` performs a palette look-up in GLSL: the sprite texture stores 8-bit indices, the `TX_SPRITEPALETTE` texture provides the final colour.
- Cinematic video also uses a palette texture (`TX_CINEMATICPALETTE`), and `ViewEntity.palette` auto-selects `SHADER_CINEMATIC` in `R_DrawEntity`.

**How it is used today:**  
Dark Reign maps and units can be loaded from `.pz2` packages. `SpriteAnimation.Palette` is set to the sprite's palette; the renderer handles the rest automatically.

### 11. UIKit Layout Improvements

**PRs:** #162, #163, #164, #194 and various smaller fixes

**What was built:**  
- `fr` (fractional) unit support for `Grid` column/row sizing (analogous to CSS `fr`).
- `UniformToFill` stretch mode in `ImageView` via a texture matrix transform.
- `PageHost` / `Page` navigation component with an `ActivePage` property.
- OpenGL **stencil buffer clipping** replacing an earlier render-texture approach for `ClipChildren`.

**How stencil clipping was implemented (PR #194):**  
`ClipChildren` now writes a full-screen stencil mask for each clipped container rather than allocating and blitting a framebuffer texture. `glStencilFunc`/`glStencilOp` gates rendering to the clipped region. The `_IsOutOfBounds` culling helper was also fixed to use the correct world-space bounds.

**How it is used today:**  
Grid layouts use `<Grid.Columns Value="1fr 2fr"/>` for proportional columns. Scroll containers use `ClipChildren="true"` for GPU-accelerated stencil clipping. `PageHost` drives tab bars and wizard flows.

### 12. Property States & Testing Infrastructure

**Commits:** `4925467` → current `tests/`

**What was built:**  
A property state system that allows components to have discrete named states (analogous to CSS pseudo-classes), and a C-level test harness in `tests/` that exercises property lookup, component registration, and message dispatch.

**How it was implemented:**  
Properties can be tagged with state metadata. `OBJ_FindShortProperty` and `OBJ_FindLongProperty` accept an optional state name to retrieve state-specific default values. Test files in `tests/` call engine functions directly through a `#include` of the relevant headers and assert expected behaviour with a lightweight macro framework.

**How it is used today:**  
Button `Hover`, `Pressed`, and `Disabled` visual states are driven by property states. The test suite is run in CI with `make test`.
